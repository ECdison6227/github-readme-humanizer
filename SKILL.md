# GitHub README 去 AI 味撰写与开源准备

## 触发条件

- 用户说"帮我写 README"、"准备推 GitHub"、"把项目开源"
- 用户提到 README "太假"、"没人味"、"AI 味太重"、"要中英文"
- 用户要求"去掉 AI 味"、"写得像人写的"、"要有真实案例"
- 用户要求整理仓库目录、准备示例文件、生成 banner 或效果对比图

## 核心原则：去 AI 味 + 有网感

好的 GitHub README 不是产品说明书，是一个真实的人在给另一个真实的人讲：

> "我遇到了什么问题，怎么解决的，你能不能直接用，哪些地方会踩坑。"

同时它也要有网感：不是抖机灵，而是让读者愿意往下看、愿意 star、愿意转发。清爽的逻辑图、一两句人话提示、不端着，都是网感。

写之前先自检这 12 条，有任何一条不满足就重写：

| # | 检查项 | 反例（AI 味） | 正例（人味） |
|---|--------|--------------|--------------|
| 1 | 开头必须有"为什么" | "本项目是一个高效的 PDF 处理工具" | "我老婆是初中数学老师，每次从作业帮导出错题本 PDF 都要截图半小时" |
| 2 | 功能描述要有具体场景 | "支持多种 PDF 类型智能识别" | "原始作业帮 PDF 自动去 logo，已处理讲义自动去页眉页脚" |
| 3 | 不能只有"能做什么" | 只列功能清单 | 必须同时列出"不能做什么" |
| 4 | 案例必须具体到文件/命令/输出 | "可以处理跨页题目" | "输入 examples/输入/数学讲义-20260616.pdf，输出 15 张 PNG" |
| 5 | 图片必须真实 | 用网络图或占位图 | 必须跑真实样例，生成 banner 和效果对比图 |
| 6 | 原理要说人话 | "采用全局标记序列策略" | "全局排序所有题号，看下一页第一个题号离页眉有多远" |
| 7 | 必须写踩坑记录 | 没有 | 列出每个版本的 bug、现象、原因、修复 |
| 8 | 限制要诚实 | "适用于所有场景" | "扫描版 PDF 不支持，多栏排版没处理" |
| 9 | 致谢要具体 | "感谢开源社区" | "感谢我老婆提供源源不断的测试题单和真实吐槽" |
| 10 | 避免空泛形容词 | "强大的、智能的、高效的、优雅的" | 用具体数据和行为替代 |
| 11 | 要有网感提示 | "请阅读以下文档以了解使用方法" | "不会用？直接把仓库地址丢给 AI，让它帮你装、教你跑" |
| 12 | 联系方式要显眼 | 把邮箱藏在末尾小字 | 用 📮 邮箱：xxx 单独一行，或在限制部分引导"小众赛道发邮件" |

## 禁用词库

写 README 时如果脑中冒出这些词，先删掉再想怎么替换：

- 强大的 / 高效的 / 智能的 / 优雅的 / 轻量级的
- 一站式 / 全方位 / 革命性 / 突破性 / 极致
- 旨在 / 致力于 / 赋能 / 助力 / 打造
- 丰富的 / 灵活的 / 完善的 / 卓越的
- 极易 / 只需 / 一键（除非真的是一键）

替换方法：把形容词换成数据、命令、场景或对比。

## 强制仓库结构

开源前，仓库里必须包含这些文件（按本次经验）：

```
repo/
├── README.md              # 中文版，主入口
├── README.en.md           # 英文版
├── LICENSE                # MIT / 其他
├── .gitignore
├── .editorconfig          # 可选但推荐
├── assets/
│   ├── banner.svg         # 项目 banner，优先 draw.io 流程图
│   └── banner.drawio      # banner 源文件，方便后续修改
├── examples/              # 示例输入/输出（ skill / 工具类项目）
├── AGENTS.md              # 给 AI Agent 的项目指引（Agent 仓库推荐）
├── CONTRIBUTING.md        # 贡献指南
├── SECURITY.md            # 漏洞报告策略 + 联系方式
├── CHANGELOG.md           # 版本变更
├── crop_questions.py      # 主代码
└── SKILL.md               # 如果是 skill 项目，说明调用方式
```

非代码项目（如 macOS App）可调整 `examples/` 为 `screenshots/` 或 `guide-video/`，但 README 仍需真实截图。

## 生成流程

### 第一步：读取项目

1. 用 LS / Glob 列出仓库根目录所有文件
2. 读取主代码文件（如 crop_questions.py），理解核心逻辑、常量、输入输出
3. 读取项目已有的 SKILL.md 或文档
4. 确认有没有示例文件、效果图、banner

### 第二步：收集真实痛点（严禁 AI 编造）

在写“为什么要做这个”之前，**必须**先问用户：

> 你做这个项目最真实的触发场景是什么？具体是谁、在什么情况下、遇到了什么麻烦？

规则：

- 用户提供了真实故事 → 直接用它作为 README 开头。
- 用户没有提供 / 说“随便写一个” → 用占位符 `[你的真实故事：...]` 标记，并在 README 里注明“此段为占位符，作者可替换为自己的真实经历”。
- **绝对禁止** AI 编造“我老婆/我老公/我朋友/我老板”等具体身份和情节。
- 示例文件里的故事可以虚构，但必须在文件顶部或 README 中明确标注“示例，非真实经历”。

### 第三步：生成 banner

**优先方案：draw.io 流程图风格 SVG**

- 用 SVG 手绘或用 draw.io 导出流程图
- 必须包含：项目标题、核心模块、模块之间的关系箭头、使用场景/输出
- 风格：浅色网格背景、圆角矩形、彩色描边、统一阴影
- 保存为 `assets/banner.svg`，并保留源文件 `assets/banner.drawio`
- 完成后用视觉模型或浏览器预览检查：箭头是否错位、文字是否截断、模块是否对齐

**备选方案：AI 生图**

- 仅当 draw.io 流程图无法表达项目气质时使用
- prompt 必须真实、具体
- 保存到 `assets/banner.png`
- 尺寸建议 landscape_16_9 或 landscape_4_3
- 如果生成效果太丑，必须退回 draw.io 方案

### 第四步：准备示例

- 找一份真实输入文件，放到 `examples/输入/`
- 运行主程序，生成输出
- 挑选最具代表性的 2-4 个效果，生成对比图放到 `examples/效果对比/`
- 命名规则：`<问题类型>_第<X>题.png`
- 如果输入文件受版权或隐私限制，无法直接上传，要在 README 里说明
- 示例里的“故事”如果为了演示而虚构，必须在文件顶部或 README 中标注“示例，非真实经历”

### 第五步：撰写 README.md（中文）

必须包含以下章节，顺序固定：

1. **banner 图**（顶部居中）
2. **一句话副标题**（说明做什么，不超过一行）
3. **为什么要做这个**（用户提供的真实痛点；AI 禁止编造。如果用户未提供，用占位符并标注）
4. **它能干什么 / 不能干什么**（双栏表格）
5. **快速开始**（安装 + 运行命令）
6. **完整案例**（具体文件名、命令、输出结构、报告片段）
   - 子章节：3 个最容易出错的场景 + 对比图
7. **工作原理**（5 步以内，说人话）
8. **关键配置**（常量表格 + 正则片段）
9. **我们踩过的坑**（版本迭代故事）
10. **限制与后续**（诚实列限制，欢迎 PR）
11. **致谢**（具体的人或项目）
12. **交流和反馈**（显眼联系方式）
13. **License**

### 顶部导航与一键 Prompt

README 顶部必须包含：

```markdown
<p align="center">
  <a href="README.md">中文</a> | <a href="README.en.md">English</a>
</p>
```

语言切换必须放在 banner 上方第一行。

紧跟 banner 后，必须加入"一键使用"Prompt 块：

```markdown
## 一键使用

如果你不想手动安装，直接把下面这段 prompt 发给你的 Coding Agent（Trae / Codex / Claude Code），它会自动完成安装并解释用法：

```text
请帮我安装并配置 https://github.com/<user>/<repo>：

1. 先 git clone 到临时目录
2. （skill 项目）运行 ./install.sh；（app 项目）运行 ./build.sh 或安装依赖
3. 告诉我这个项目是做什么的、核心功能/触发词是什么
4. 如果需要初始化配置（个人资料、偏好设置、权限），引导我完成
5. 用一个简单例子演示如何调用/使用
```
```

英文版对应 `One-click Setup`。

### 隐私检查

推送前必须扫描仓库，确保不包含：

- 真实姓名、身份证号、手机号、家庭地址
- 真实学生/客户数据
- 个人私钥、密码、API key
- 公司内网地址、私有仓库地址
- 本地绝对路径（如 `/Users/edison/...`）

如发现敏感信息：

- 立即从当前工作区删除
- 如果已经 commit 且 push，使用 `git filter-repo` 或 BFG Repo-Cleaner 从历史中彻底移除，然后 force push
- 在 README 中说明哪些数据是示例/占位符

### 仓库命名与描述

给用户提供 2-4 个命名方案：

| 方案 | 仓库名 | 描述 | 适用场景 |
|------|--------|------|----------|
| A | `xxx-skills` | 自用 AI 编程 Skill 集合 | skill 仓库 |
| B | `xxx-cropper` / `xxx-processor` | 具体工具描述 | 工具类 |
| C | `xxx` | 简洁品牌名 | 产品/App |

仓库描述要：

- 一句话讲清价值
- 不超过 GitHub 描述长度限制
- 不要出现"去 AI 味"等内部黑话

### 提交规范

commit message 不要写内部术语：

- ❌ "去 AI 味 README"
- ❌ "按用户要求改一下"
- ✅ `refactor: rewrite README with real-world cases and pitfalls`
- ✅ `fix: connect banner arrow to output nodes`
- ✅ `docs: add SECURITY.md and CONTRIBUTING.md`

推送前检查 `git config user.name` 和 `user.email` 是否匹配 GitHub 账号，否则 commit author 会显示为本地机器名。

### 第六步：撰写 README.en.md（英文）

- 不是中文直译，是按英文开源社区习惯重写
- 保持第一人称、但必须是用户提供的故事，AI 禁止编造
- 章节与中文一致
- 避免中式套话，如 "This project is a powerful tool"

### 第七步：最终审查

用下面的检查清单逐条过：

- [ ] README.md 和 README.en.md 都已存在
- [ ] 顶部有语言切换 `中文 | English`，且放在 banner 上方第一行
- [ ] 有"一键使用"Prompt 块，可直接复制给 Trae/Codex/Claude Code
- [ ] banner 是 draw.io 风格 SVG，箭头无错位，文字无截断
- [ ] 保留 banner 源文件 `assets/banner.drawio`
- [ ] 开头"为什么要做这个"来自用户真实痛点；若未提供，已用占位符标注
- [ ] 没有 AI 编造的"我老婆/我朋友/我老板"等具体身份和情节
- [ ] 示例文件里的虚构故事已标注"示例，非真实经历"
- [ ] 同时列出"能做什么"和"不能做什么"
- [ ] 快速开始有具体可复制的命令
- [ ] 完整案例包含真实文件名、命令、输出
- [ ] 有 2-4 张效果对比图或真实截图，且文件存在
- [ ] 原理部分没有空话，有具体数字/阈值
- [ ] 踩坑记录至少 3 条，每条有现象+原因+修复
- [ ] 限制部分诚实，没有"适用于所有场景"
- [ ] 没有出现禁用词库中的空泛形容词
- [ ] 英文版不是直译，符合英文社区表达
- [ ] 有至少 1 处"网感提示"（如"把仓库地址丢给 AI"）
- [ ] 联系方式用显眼格式单独列出
- [ ] 已做隐私检查，无真实个人/学生/公司敏感信息
- [ ] AGENTS.md / CONTRIBUTING.md / SECURITY.md / CHANGELOG.md 已补充（Agent/工具类项目）
- [ ] commit message 没有"去 AI 味"等内部黑话
- [ ] git config user.name / user.email 已匹配 GitHub 账号

### 第八步：推送准备

1. 检查 `.gitignore` 是否忽略了临时文件、虚拟环境、输出目录
2. 检查 `LICENSE` 是否存在
3. 检查示例文件是否已放入仓库
4. 检查 analyze.py 等调试脚本是否已删除（用户未要求保留则删除）
5. 建议用户手动处理受限制文件（如 Desktop 路径、大文件）
6. 用户确认后执行：

```bash
git init  # 如果还没初始化
git add README.md README.en.md LICENSE .gitignore assets/ examples/ <主代码文件>
git commit -m "init: add PDF question cropper with human-style README"
git remote add origin <用户提供的仓库地址>
git push -u origin main
```

## 人味 + 网感写法示例对比

### 开头

**AI 味：**

> 本项目是一个基于 PyMuPDF 的高效 PDF 题目裁剪工具，旨在帮助用户自动拆分 PDF 中的题目并生成独立图片。

**人味：**

> 我老婆是初中数学老师。每次从作业帮导出错题本 PDF，她都要把每道题单独截图出来，拼成给学生上课用的讲义。一份 15 题的卷子，她要切半小时。

### 网感提示

**端着：**

> 请阅读以下文档以了解使用方法。

**有网感：**

> 不会用？直接把本仓库地址 `https://github.com/xxx/xxx` 丢给 AI，让它帮你装环境、改配置、跑起来。

### 联系方式

**藏着：**

> 如有问题请联系作者。

**显眼：**

> 📮 邮箱：**2014184720@qq.com**

### 功能描述

**AI 味：**

> 支持智能识别多种题目格式，实现精准裁剪。

**人味：**

> 第 4 题题号在第 2 页底部、正文在第 3 页顶部，能自动拼成一道图。

### 原理说明

**AI 味：**

> 本工具采用全局标记序列策略进行跨页判断，并通过内容边界保护机制优化裁剪结果。

**人味：**

> 全局排序所有题号，看下一页第一个题号离页眉有多远。如果它贴在页顶，说明是下一题；如果它离页顶很远，说明是上一题的续页。

## 踩坑记录写法模板

每条踩坑记录必须包含：

```markdown
### 第 X 版：一句话概括问题

**现象**：用户看到了什么错误输出
**原因**：为什么代码会这样
**修复**：改了什么，现在怎么样了
```

示例：

```markdown
### 第一版：每道题被拆成标题和正文两块

**现象**：15 题切出了 30 张图
**原因**：`get_text("blocks")` 把"第 1 题"和正文分成两个 block
**修复**：改成 `get_text("dict")` 的 line/span 级别，把同一行的 span 合并后再匹配
```

## 与用户协作方式

1. **先生成再确认**：先写出 README 初稿，告诉用户我准备怎么写，不直接推送
2. **让用户看到布局**：展示章节结构和示例图
3. **用户说可以后再推送**：用户之前明确说"这个可以推送"，也要先确认一遍
4. **无法自动处理的手动项要明确列出**：
   - 示例 PDF 需要手动复制到 examples/输入/
   - 调试脚本需要手动删除
   - 仓库地址需要用户提供
   - **“为什么要做这个”的真实痛点必须由用户提供，AI 不能代编**

## 常见错误

- 把 README 写成技术文档而不是项目介绍
- 只写中文不写英文，或英文是机器直译
- 用占位图、网络图或 AI 生成的抽象图代替真实效果图
- 只写优点不写限制
- 缺少具体数字（阈值、页数、题目数）
- 踩坑记录写成 changelog 而不是故事
- 逻辑图用单调箭头，没有清爽感或人味标注
- 联系方式藏在小字里，读者找不到
- **AI 编造“我老婆/我朋友/我老板”等具体身份和情节，而不是先问用户真实痛点**
